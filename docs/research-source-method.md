# 外部信息调研、校准与信源分级方法

- 状态：内部方法文档
- 目标：为本仓库中所有强时效、外部事实型文档提供统一的搜索、验证、多语言校准与信源分级规则
- 适用对象：
  - 价格、政策、支持地区、可用性、配额、套餐边界
  - 第三方工具、宿主、API、渠道与市场现象
  - 需要多语言 deep research 的外部调研文档
- 相关文档：
  - 方法应用示例见 [how-to-get-ai.md](/data/projects/x-promptkit/docs/how-to-get-ai.md)
  - 如果关注 Skill / Capability 工具链选择，见 [skill-dev-tool-options.md](/data/projects/x-promptkit/docs/skill-dev-tool-options.md)

## 1. 这篇文档回答什么

这篇文档不是调研结果本身，而是“**怎么做调研**”的方法说明。

它要回答的是：

- 遇到高时效的外部事实，应该怎么搜索
- 搜到来源后，应该怎么验证
- 多语言 deep research 应该怎么做，做到什么程度
- 不同类型的信源应该怎么分级、上色、使用
- 证据不够时，应该怎么降级写法，而不是硬写结论

这篇文档不回答：

- 开户、支付、注册、代理或中转的实操
- 灰色渠道的执行细节
- 私域情报获取方式
- 企业法务或采购流程

## 2. 搜索技巧

### 2.1 搜索顺序

默认按这个顺序搜，不要一开始就跳论坛：

1. 官方文档 / 官方帮助中心 / 官方定价页
2. 官方产品页 / 官方公告 / 官方博客
3. 主流媒体或云厂商公开页面
4. 目录、索引、汇编型页面
5. 社论、专栏、评论
6. BBS、论坛、帖子

这套顺序的目的不是“只信官方”，而是先把**可被精确引用的事实边界**找出来，再看外部讨论。

### 2.2 默认搜索策略

- 先搜“对象类型”，再搜“品牌名”。
  - 先确认它是 `订阅`、`API`、`CLI`、`宿主`、`路由层`、`云市场` 还是 `灰色渠道形式`
- 先搜“规则页”，再搜“评价页”。
  - `pricing`
  - `help`
  - `supported countries`
  - `availability`
  - `terms`
  - `limits`
  - `extra usage`
  - `rate limits`
- 先搜“主语言原文”，再搜“多语言本地化页”。
- 对灰色玩法，只搜“存在性”“叫法”“社区报告”，不搜执行步骤。

### 2.3 可复用 query 模板

下面这些模板应优先复用，再按对象名替换：

```text
site:<official-domain> <brand> pricing
site:<official-domain> <brand> help

site:<official-domain> <brand> availability
site:<official-domain> <brand> extra usage

```

```text

```

```text

```

```text

```

### 2.4 时间锚定搜索

强时效问题必须显式带时间锚：

- 年份：`2026`
- 事件日期：例如 `2026-04-05`
- 相对日期转绝对日期：不要只写“最近”“今天”“上周”

适用场景：

- 调价
- 政策更新
- 限量发售 / 售罄
- 封禁 / 解除封禁
- 支持地区变化

### 2.5 反向搜索与交叉搜索

如果正向搜索只找到转述，继续做这几步：

- 用转述中的关键短语回搜官方页
- 用产品名 + 计划名 + 日期回搜
- 用 URL slug 反推帮助中心文章
- 用“品牌 + 事实 + language code”回搜多语言页
- 用“品牌 + 事实 + cached / archive / mirror”确认页面曾存在，但不要把缓存页当现行政策

## 3. 验证技巧

### 3.1 五层验证

搜到信息后，默认至少做下面五种验证：

1. 事实验证  
   这件事是否真的发生，还是二手转述。

2. 时间验证  
   这是不是当前有效信息，是否已经过时。

3. 边界验证  
   它适用于哪个产品、哪个计划、哪个宿主、哪个区域、哪个 API。

4. 来源验证  
   这条说法来自谁，是否值得直接支撑正文判断。

5. 冲突验证  
   如果不同来源说法不一致，哪一层来源优先，哪些内容必须降级写法。

### 3.2 必查边界

外部信息最容易错在“边界没拆开”。默认要查这些：

- 第一方产品 vs 第三方宿主
- 聊天订阅 vs API
- 官方 CLI / agent vs 第三方 coding 宿主
- 账号复用 vs API 复用
- 支持地区 vs 可访问页面
- 上线过 vs 当前仍可用

### 3.3 冲突处理规则

当来源冲突时，按下面顺序处理：

1. 官方现行页面优先于媒体转述
2. 原始定价 / 帮助页面优先于产品介绍页
3. 最新日期优先于旧日期
4. 主语言原文优先于本地化页面
5. `A/B` 级优先于 `C/D/E`

如果冲突仍无法消除，不要硬下结论，改写成：

- `目前只能确认……`
- `不同来源存在冲突……`
- `尚不足以确认……`
- `至少可以确认……，但不能据此推出……`

### 3.4 什么时候不该写

出现下面任一情况，应删除、降级或显式标不确定：

- 只有单条 `D/E` 级来源
- 只有社区叫法，没有对象边界
- 只有“有人说被封”，没有对应政策或多来源交叉
- 只有缓存页，没有现行页
- 只有截图转述，没有原始页面

## 4. 信源分级方法

### 4.1 分级与颜色

- <span style="color:#2e7d32;"><strong>A</strong></span>：官方文档、官方帮助中心、官方定价页
- <span style="color:#1565c0;"><strong>B</strong></span>：主流媒体、云厂商官方博客、主流公开产品页
- <span style="color:#ef6c00;"><strong>C</strong></span>：社区目录、项目索引、公开资料汇编
- <span style="color:#c62828;"><strong>D</strong></span>：社论、专栏、评论、自媒体分析
- <span style="color:#6a1b9a;"><strong>E</strong></span>：BBS、论坛、帖子、经验贴

### 4.2 每一级能支撑什么

- `A`
  - 可直接支撑正文主判断
  - 可用于价格、政策、支持地区、产品边界、官方可用性
- `B`
  - 可补充主判断
  - 可用于现象、市场动作、产品公开发布、公开采访
- `C`
  - 只能证明“这种对象或项目存在过”
  - 不能单独证明价格、政策、封禁规则仍然有效
- `D`
  - 只能证明“市场里有人这样分析或评论”
  - 不能单独证明事实成立
- `E`
  - 只能证明“社区里有人这么说、这么叫、这么报告”
  - 不能单独证明稳定性、政策、封禁和价格

### 4.3 越级使用错误

这些都是应避免的：

- 用论坛帖证明官方价格
- 用评论文章证明平台政策原文
- 用目录站证明某服务当前仍可用
- 用社论证明某玩法一定会封
- 用单个用户帖证明“普遍现象”

## 5. 多语言校准方法

### 5.1 默认语言集合

除非任务另有要求，默认至少交叉看：

- 中文
- 英文
- 日语
- 法语
- 俄语
- 越南语
- 印尼语
- 印地语

### 5.2 多语言页的用途

多语言页主要用于：

- 验证本地化覆盖是否存在
- 验证高层事实在不同语言中是否一致
- 观察厂商是否真的在某语言市场做公开分发

多语言页不应用于：

- 替代英文主文作为最终政策依据
- 单独证明价格、支持地区、条款一定最新
- 覆盖掉英文主文中的边界说明

### 5.3 多语言校准步骤

建议固定按这 5 步做：

1. 找英文或主语言原文
2. 找对应语言页面或本地化站点
3. 对齐同一个事实点
4. 记录“有本地化 / 无本地化 / 本地化不完整”
5. 回到主语言原文下最终判断

### 5.4 多语言页常见问题

- 页面存在，但只是首页，不是帮助中心
- 页面有本地化，但内容滞后
- 页面有翻译，但 slug 不稳定
- 页面是机器翻译，不适合精确引用条款
- 某语言完全没有同等级页面

这种情况下推荐写法：

- `未找到同等级本地化帮助中心页`
- `仅找到套餐页，未找到对应帮助中心页`
- `找到本地化入口，但未找到同等级政策页`

## 6. 输出规范

### 6.1 文档头部必须写

对强时效文档，头部至少要有：

- 高时效性提示
- 调研日期
- 相关文档
- 信源分级规则

### 6.2 正文必须写的边界

如果文档涉及高争议或灰色对象，正文必须补：

- 证据边界说明
- 哪些内容只来自 `D/E`
- 哪些判断只到“存在性”层级
- 哪些结论已由 `A/B` 支撑

### 6.3 推荐句式

可直接确认时：

- `可以确认……`
- `官方页面明确写明……`
- `按 <绝对日期> 的官方页面……`

高概率但未完全确认时：

- `高概率属实，但目前只找到……`
- `至少可以确认……，但不能据此推出……`
- `目前没有足够证据确认……`

只见于低等级来源时：

- `仅见于 D/E 级来源`
- `社区里存在这种说法`
- `这更像市场现象，而不是已完成官方验证的事实`

### 6.4 输出前检查清单

- 是否明确写了调研日期
- 是否区分了官方与非官方来源
- 是否把多语言页当成了主文替代品
- 是否把论坛或社论越级写成结论
- 是否对证据不足内容做了降级写法
- 是否避免了“今天、最近、最新”这类无日期表述
